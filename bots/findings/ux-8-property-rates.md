## Findings

| # | Severity | Area | File:line | Issue | Recommendation |
|---|----------|------|-----------|-------|----------------|
| 1 | High | a11y — sortable table | `app/rates/RateBoardClient.tsx:234–259` | Sortable `<th>` columns (Provider, Rate, Min Deposit, Rating) have no `aria-sort` attribute. Screen readers cannot announce current sort direction or that columns are interactive. No `scope="col"` on any `<th>`. No `<caption>` on the table. | Add `scope="col"` to all `<th>`, add `aria-sort="ascending"/"descending"/"none"` dynamically based on `sortField`/`sortDirection`, add `<caption className="sr-only">Savings and term deposit rates</caption>`. |
| 2 | High | a11y — tab semantics | `app/rates/RateBoardClient.tsx:196–222` | The two tab buttons ("Savings Accounts", "Term Deposits") use plain `<button>` elements with no `role="tab"`, no `role="tablist"` wrapper, no `aria-selected`, and no `aria-controls` / `id` linkage to the table panel. Keyboard users and AT cannot navigate them as tabs. | Wrap in `<div role="tablist">`, add `role="tab"` + `aria-selected={activeTab === 'savings'}` to each button, add `id` + `aria-controls` linking to the table region wrapped with `role="tabpanel"`. |
| 3 | High | Stale-rate freshness label | `app/rates/RateBoardClient.tsx:160–164, 387` | `lastUpdated = new Date()` is evaluated at **client render time** (or hydration), not at the time rates were actually fetched. With `revalidate = 3600` on the parent page, the displayed "Last updated: [today]" date will be wrong for up to an hour and misleads users about data freshness. On a cached ISR response served at 11:59 pm, the label will show the date it was served, not when rates were confirmed. | Pass a `fetchedAt: string` prop from the server component (capture `new Date().toISOString()` at the point the Supabase query returns in `page.tsx`) and display that value in the client. |
| 4 | High | IA — broken navigation dead-end | `app/rates/RateBoardClient.tsx:305` | Provider links in the rates table go to `/broker/${provider.slug}`. However `app/rates/today/page.tsx:115–117` routes the same savings/term-deposit providers to `/savings/${broker_slug}` — an inconsistent and likely broken path (`/savings/[slug]` does not appear to be a valid route; the vertical page is `/savings`). Both patterns may 404 for real products. | Decide on one canonical provider detail path. If `/broker/[slug]` is correct, fix `rates/today`. If a dedicated savings provider page exists (`/savings/[slug]`), link the rates board there too. Add a route-existence check in CI. |
| 5 | High | Directory-primitive reuse | `app/rates/RateBoardClient.tsx` (whole file) | The `/rates` rate board is a bespoke table that replicates sort, tab, result-count, and empty-state patterns already provided by `components/directory/{SortDropdown, TabBar, ResultCount, EmptyState}`. The same primitives are used in `/compare/super`, `/compare/etfs`, and `/advisors`. This is exactly the divergence called out in CLAUDE.md. | Replace bespoke tab buttons with `<TabBar>` from `components/directory/TabBar.tsx`, replace result-count footer text with `<ResultCount>`, replace sort arrow spans with `<SortDropdown>` or `aria-sort` on `<th>` (the latter is preferred for a `<table>`), replace empty-state `<td>` with `<EmptyState>`. |
| 6 | Medium | a11y — contrast | `app/rates/RateBoardClient.tsx:386–393` | "Last updated" and result-count footer text uses `text-[0.62rem]` (9.92 px) with `text-slate-400` (approx. #94a3b8) on white — fails WCAG AA for normal text (contrast ratio ~3.2:1; minimum is 4.5:1 for text below 18px). The same micro-text pattern (`text-[0.56rem]`, `text-[0.62rem]`) recurs in `mortgage-calculator/MortgageCalculatorClient.tsx` and `savings-calculator/SavingsCalculatorClient.tsx`. | Increase to `text-xs` (12px) minimum and use `text-slate-500` or darker, or treat as decorative and make it `aria-hidden`. |
| 7 | Medium | a11y — tap targets | `app/rates/RateBoardClient.tsx:372` and `app/mortgage-calculator/MortgageCalculatorClient.tsx:232,254` | The "Visit Site →" / "View Details →" action buttons in the rates table are `px-3 py-1.5` (approx. 30px tall). The quick-picker preset buttons in mortgage/savings calculators are `text-[0.56rem] px-2 py-1` (approx. 24px tall). Both are below the WCAG 2.5.8 recommended 24×24px minimum and iOS/Android 44px minimum. | Increase action buttons to `min-h-[44px]` or `py-2.5`. Increase preset pills to `py-1.5 min-w-[44px]`. |
| 8 | Medium | IA — cross-link gap | `app/rates/RateBoardClient.tsx` (whole file) | The `/rates` board has no cross-link to `/rate-alerts` (sign up for alerts), `/rates/today` (what changed today), or `/savings-calculator` (how much would I earn?). These are the three most obvious next-step pages for a user who has just compared rates. `/rates/today` also has no link back to the main `/rates` board — only `← Rate board` text anchor, with no discovery path *to* `/rates/today` from the main rate board. | Add a secondary action bar below the rates table with links to `/rate-alerts`, `/rates/today`, and `/savings-calculator`. Add a `href="/rates/today"` link from the "Last updated" row in the rates board footer. |
| 9 | Medium | Visual consistency — inline styles | `app/rates/today/page.tsx:82–171` | The `/rates/today` page uses 34 inline `style={}` attributes (custom CSS vars, px values) instead of Tailwind classes, diverging completely from the Tailwind-first convention of every other page in this domain. Remove `iv2-card` if it is not in the active design system. |
| 10 | Medium | Forms — missing label association | `app/mortgage-calculator/MortgageCalculatorClient.tsx:220–228` | The "Loan amount" and "Interest rate" inputs use `<label className="block ...">Loan amount</label>` followed by a `<div>` wrapper and then the `<input>` — the label is not programmatically associated to the input (no `for`/`htmlFor` attribute, no wrapping label). Screen readers will not announce the field name when the input receives focus. Same pattern in `savings-calculator/SavingsCalculatorClient.tsx:139–145` and `property-yield-calculator/PropertyYieldCalculatorClient.tsx:211–218`. | Add `htmlFor="loan-amount"` to the `<label>` and `id="loan-amount"` to the `<input>`, or wrap the input inside the label element. | 11 | Medium | a11y — amortization schedule and rate comparison use divs-as-table | `app/mortgage-calculator/MortgageCalculatorClient.tsx:404–418` | The amortization schedule (Year / Principal / Interest / Balance) is rendered as a `grid grid-cols-4` of `<div>` rows with a header row of `<span>` labels. The rate comparison section (lines 363–393) has the same problem. | Replace both with `<table>`, `<thead>/<tbody>`, `<th scope="col">` for headers, `<td>` for data cells. | 12 | Medium | IA — `/mortgage` and `/home-loans` overlap without clear hierarchy | `app/mortgage/page.tsx:1–210`, `app/home-loans/page.tsx:1–160` | Both `/mortgage` and `/home-loans` are hub-level pages covering the same topic. `/mortgage` has a full guide with FAQs; `/home-loans` uses the `HubPage` component with a service grid. They cross-link to each other via the navigation but a user arriving at one may not realise the other exists or which is canonical. Breadcrumbs in sub-pages (e.g. `/home-loans/variable`) use "Home Loans" pointing to `/home-loans`, but the mortgage page at `/mortgage` also claims to be the guide hub. | Establish one canonical hub (suggest `/home-loans`) and make `/mortgage` a redirect or content entry-point that clearly defers to `/home-loans`. | 13 | Medium | Microcopy — comparison rate disclosure flag (REGULATORY) | `app/mortgage-calculator/MortgageCalculatorClient.tsx:474` | The mortgage calculator's SEO content mentions "comparison rate" as a concept to check but the calculator itself only shows headline rate scenarios. This calculator does not display or advertise specific product rates, but any future addition of live lender rates to this page would trigger the comparison rate requirement. **FLAG for legal review before adding live product rates to the calculator.** | Current state (generic rate input, no named products) appears compliant; flag this as a hard constraint for any feature that would add real lender rates or named products to calculator output pages. ## Quick-wins (safe to auto-apply)

1. **`scope="col"` on all `<th>` in `app/rates/RateBoardClient.tsx`** — one-line addition per header cell, no logic change. Also add `<caption className="sr-only">`.
2. **`aria-label` on star-rating span** (`app/rates/RateBoardClient.tsx:86`) — add `role="img"` and `aria-label={`${rating} out of 5 stars`}` to the returned `<span>`.
3. **Expand "P&I" abbreviation** (`app/mortgage-calculator/MortgageCalculatorClient.tsx:283`) — add `title="Principal and Interest"` to the button.
4. **Fix `lastUpdated` to be a server-side prop** — capture `new Date().toISOString()` in `page.tsx` and pass as `fetchedAt` to `RateBoardClient`.
5. **Add `htmlFor` / `id` wiring** on calculator inputs — affects `mortgage-calculator`, `savings-calculator`, `property-yield-calculator` (finding #10).
6. **Remove hardcoded rate teasers** from `app/property/page.tsx:591–599` and replace with "Compare current rates" CTA without specific numbers.

## Healthy patterns

- **`RateAlertSignupForm.tsx`** is an excellent a11y example: proper `<fieldset>`/`<legend>` for radio-style button groups, `aria-pressed` on toggle buttons, `role="alert"` on error messages, `role="status"` on success/token states, honeypot visually hidden with `aria-hidden="true"`, `noValidate` with custom error handling, `autoComplete="email"`.
- **Rate comparison ISR** (`revalidate = 3600` across `/rates`, `/savings`, `/term-deposits`) is correctly set — not too stale, not causing unnecessary re-renders.
- **Mortgage calculator cross-calculator state persistence** via `useCalculatorState` hook and `useUrlSync` for shareable deep links is a well-built feature.
- **`app/home-loans/variable/page.tsx`** demonstrates thorough NCCP-aware compliance: clear "general information only / not credit assistance" disclosures per section, CTA to licensed broker, and `GENERAL_ADVICE_WARNING` in footer.
- **`app/rate-alerts/page.tsx`** has complete breadcrumb JSON-LD, FAQ JSON-LD, WebPage schema, and all compliance copy drawn from `lib/compliance.ts` — good single-source-of-truth discipline.
- **`app/property/page.tsx`** uses `Promise.all` for parallelised DB fetches and graceful `try/catch` degrade — no 503 on partial DB failure.
