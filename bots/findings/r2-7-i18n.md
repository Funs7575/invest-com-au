# R2 Audit — Internationalization

## How i18n works here

The repo has four registered locales (`en`, `zh`, `ko`, `ar`) declared in `lib/i18n/locales.ts`. The canonical English locale is `en-AU`; the others are `zh-CN`, `ko-KR`, and `ar-AE`. The direction flag `LOCALE_DIR` marks only `ar` as RTL.

**Routing:** Localised pages live under `app/zh/`, `app/ko/`, and `app/ar/` Next.js route segments. There is no Next.js `[locale]` dynamic segment or middleware redirect — locale is inferred from the URL prefix by `stripLocalePrefix()` in `proxy.ts` / root layout.

**Root layout RTL/lang stamping:** `app/layout.tsx` reads `x-pathname` (stamped by `proxy.ts` at line 68) via `headers()`, calls `stripLocalePrefix`, then sets `<html lang={htmlLang} dir={htmlDir}>`. For any `/ar/*` path this correctly yields `lang="ar-AE" dir="rtl"`.

**Dictionaries:** `lib/i18n/dictionaries.ts` holds `ForeignInvestmentDict` (hub) and `SubPageDict` (SIV, Property, Tax) for `en`/`zh`/`ko`. Arabic has **no** entry in either dictionary; calls to `getForeignInvestmentDict("ar")` / `getSubPageDict(slug, "ar")` silently fall back to English. The Arabic pages use their own bespoke translation objects (`AE_CONFIG_AR`, `SA_CONFIG_AR`) in `lib/i18n/translations/`.

**Language switcher:** Inline in `ForeignInvestmentLocalizedPage` and `ForeignInvestmentSubPage` — no standalone component. Iterates `LOCALES` to emit pill links.

**hreflang:** ZH/KO hub and sub-pages emit `alternates.languages` in metadata. Arabic pages are marked `noindex` and have no hreflang block yet (by design, pending native reviewer).

## Quick wins

1. **Filter `ar` from language switchers for non-existent routes** (Findings 1 & 2) — single-line change to both components; eliminates live 404s immediately.
2. **Add `"/foreign-investment/saudi-arabia"` to `LOCALE_KNOWN_PATHS.ar`** (Finding 5) — one-liner in `lib/i18n/locales.ts`; fixes sitemap gap.
3. **Translate "Home" breadcrumb** (Finding 6) — add a 4-key record to `locales.ts` and swap the hardcoded string in both components.
4. **Wrap Arabic math expression in `dir="ltr"`** (Finding 10) — inline fix in `ae-ar.ts` or the UAE page template.

## Notable

- **RTL root layout stamping is correct.** `proxy.ts` stamps `x-pathname`; `app/layout.tsx` reads it and sets `<html dir="rtl">` for `/ar/*` routes. No gap in the root shell.
- **Arabic pages are intentionally `noindex`** until a native-reviewer pass lands. The lack of hreflang on AR pages is a documented, deliberate deferral (Phase 5c), not a gap.
- **No `Accept-Language` redirect exists.** There is no middleware that routes Chinese/Korean/Arabic browser users to localised pages — discovery is entirely through the inline language switcher. For the current scale (3 locales, specialist audience) this is acceptable but limits organic locale-based traffic.
- **`lib/currency.ts:formatCurrency`** accepts a `locale` parameter and passes it to `Intl.NumberFormat` — the right pattern. However, `formatAUD` (line 92) hardcodes `"en-AU"` and has no locale parameter, meaning AUD amounts on localised pages always format as Australian English (comma-separator, `$` prefix) regardless of user locale. For zh-CN users `¥`-style formatting would be more familiar.
- **No locale persistence cookie or localStorage store.** Language preference is encoded only in the URL. If a ZH user navigates to a non-localised page (e.g., broker comparison), there is no mechanism to remember their language and resurface the switcher.
