# UX/UI Audit — Super, retirement, SMSF & life events

## Scope

Routes audited: `app/super`, `app/super-contributions-calculator`, `app/smsf`, `app/smsf-calculator`, `app/retirement`, `app/retirement-calculator`, `app/fire-calculator`, `app/redundancy`, `app/aged-care`, `app/inheritance`, `app/dividends`, `app/dividend-reinvestment-calculator`, `app/franking-credits-calculator`, `app/cgt-calculator`, `app/negative-gearing`, `app/investment-income-tax-calculator`.

Calculator components inspected: `FireCalculatorClient.tsx`, `RetirementCalculatorClient.tsx`, `SMSFCalculatorClient.tsx`, `SuperContributionsClient.tsx`, `CgtClient.tsx`/`CgtCalculator.tsx`, `FrankingClient.tsx`/`FrankingCalculator.tsx`, `FrankingCalculatorClient.tsx` (dividends/calculator), `InvestmentIncomeTaxClient.tsx`, `DividendReinvestmentClient.tsx`, `NegativeGearingCalculatorClient.tsx`.

**CalculatorShell reuse status:** `CalculatorShell.tsx` is NOT used by any calculator in this domain. All standalone calculators build their own full-page layouts with per-calculator accent colours, heading sizes, and hero sections. `CalculatorShell` is only used by `components/RdTaxCalculator.tsx` (embedded on grant pages). Two layout families exist:
- **Hero-gradient + staged flow** (`RetirementCalculatorClient`, `SMSFCalculatorClient`): large coloured hero, grid inputs, explicit "Calculate" button, results appear below on click.
- **Two-column live-update** (`FireCalculatorClient`, `SuperContributionsClient`, `DividendReinvestmentClient`): inputs panel left, live results right, no submit button.
- **Hub-inline variant** (`CgtCalculator`, `FrankingCalculator` via `CalcShared.tsx`): `CalcSection` wrapper, used when embedded in `/calculators` hub.

## Healthy patterns

- **`InvestmentIncomeTaxClient.tsx`** is the best-practice reference in this domain: uses `aria-live="polite" aria-atomic="true"` on the results panel, has explicit `htmlFor`/`id` pairing via `useId()` in `MoneyField`, has an empty state (icon + message when no income entered), has a `<nav aria-label="Breadcrumb">` trail, has correct Stage 3 tax brackets in `lib/calculators/investment-income-tax.ts`, and has an inline `<form onSubmit={e=>e.preventDefault()}>` semantic wrapper.
- **Compliance footers** are consistently present via `ComplianceFooter variant="calculator"` on every calculator `page.tsx` in this domain — none missed.
- **Cross-calculator state persistence** via `useCalculatorState()` hook is implemented on all major calculators (FIRE, Retirement, SMSF, Super Contributions, DRP, Negative Gearing) — good UX for returning users.
- **Inline validation** on age inputs in Retirement Calculator (`Math.max(currentAge + 1, ...)`) prevents nonsensical retirement-age-less-than-current-age inputs.
- **Division-by-zero guard** in FIRE Calculator at `FireCalculatorClient.tsx:67` (`wr > 0 ? annualExpenses / wr : 0`) prevents NaN propagation.
- **HubPage breadcrumbs**: all hub pages (`/super`, `/retirement`, `/smsf`, `/aged-care`, `/inheritance`, `/redundancy`) use `HubPage` which auto-generates both visible breadcrumb nav and JSON-LD `BreadcrumbList` from the hub config — consistent and correct.
- **General advice disclaimer** in `lib/compliance.ts` (`GENERAL_ADVICE_WARNING`) is used consistently in all calculator pages and hub pages across the domain.
