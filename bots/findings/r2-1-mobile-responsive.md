# R2 Audit — Mobile & responsive

## Scope

Second-pass mobile/responsive audit of invest.com.au (Next.js App Router, Tailwind, TypeScript strict). Sampled: shared layout/nav (`components/layout/Navigation.tsx`, `components/LayoutShell.tsx`, `components/MobileBottomNav.tsx`), compare pages (`app/compare/`, `app/shortlist/compare/`, `app/versus/`, `app/advisors/compare/`, `app/invest/compare/`), advisor/broker profiles (`app/advisor/[slug]/`), calculators (`app/etfs/screener/`, `app/brokerage-fee-index/`), content/guide pages (`app/first-home-buyer/stamp-duty/`, `app/global-investing/currency/best-fx-providers/`, `app/tax/trusts/`), shared floating/sticky components (`StickyAdFooter`, `StickyCTABar`, `MobileFloatingCTA`, `QuizPromptBar`, `OnThisPage`, `InvestorCopilot`, `GettingStartedChecklist`, `FloatingRightCTA`, `ForeignInvestmentNav`). Did NOT repeat Round 1 a11y/IA/contrast findings.

## Quick wins (safe to auto-apply)

1. **`StickyAdFooter` dismiss button** (`components/StickyAdFooter.tsx:171`): Change `p-1` → `p-3 min-w-11 min-h-11`. One line, no logic change, immediate 44 px tap target.
2. **`InvestorCopilot` chat panel width** (`components/InvestorCopilot.tsx:105`): Add `max-w-[calc(100vw-2rem)]` to the existing className. One-token addition.
3. **Sticky table cell backgrounds** (`app/shortlist/compare/CompareClient.tsx:234,264,287,310`): Replace `bg-slate-50/50` with `bg-slate-50` and `bg-inherit` with `bg-white` on all sticky cells. Prevents iOS Safari bleed-through.
4. **Toast bottom offset** (`components/Toast.tsx:29`): Change `bottom-20` → `bottom-28` in the className string and the imperative DOM string. Two-character change.

## Notable

- **Bottom-layer crowding is the biggest systemic issue**: Up to 5 separate `fixed bottom-0` elements can co-exist simultaneously on a broker review or content page — `MobileBottomNav` (z-40), `StickyAdFooter` (z-40), `MobileFloatingCTA` (z-50), `StickyCTABar` (z-50), and `OnThisPage` mobile pill (z-40). There is no coordination layer, each component independently calculates its `bottom` value. This creates a high risk of elements overlapping each other or the device's home-indicator area on modern iPhones. A shared `--mobile-chrome-height` CSS custom property (set once by `MobileBottomNav` via a `useEffect` measurement) would solve this cleanly.
- **`overflow-x-auto` hygiene is generally good**: Most tables in guide pages correctly wrap with `overflow-x-auto` and `min-w-[Npx]`. The screener/compare tools are the outliers needing mobile card fallbacks.
- **Safe-area handling is inconsistently applied**: `BottomSheet`, `MobileBottomNav`, `MobileFloatingCTA`, `CompareSelectionBar`, and `StickyCTABar` all handle `safe-area-inset-bottom` via different mechanisms (CSS class `safe-area-inset-bottom`, `env()` in `style`, CSS calc). Centralise in a utility class.
