## Layouts that need explicit `dir="ltr"` (don't flip)

1. **All numeric / currency tables and stat blocks.** Numbers stay LTR even inside RTL prose. Apply `dir="ltr"` on the `<table>` or stat-value wrapper for `30%`, `AUD 5,000`, `12.5%`, etc.
2. **Code / monospace strings.** Any `font-mono` block (ABNs, transaction reference IDs, URLs displayed as text).
3. **Email + URL text rendered as content.** `finn@invest.com.au`, `https://invest.com.au/...`.
4. **`rotate-180` instances (88).** Most are accordion chevrons. The rotation is direction-agnostic; the *icon* it rotates is directional. `ChevronDown rotate-180` → safe (becomes Up). `ChevronRight rotate-180` → needs RTL handling.
5. **`translate-x-N` slide-in modals/drawers/toasts (~50 instances).** Slide direction defined; in RTL should slide from opposite side. Use `rtl:translate-x-N` mirror utilities.
6. **Brand wordmarks.** "invest.com.au" should not render as "ua.moc.tsevni" — `dir="ltr"`.
7. **The 1,165 `Chevron*` / `Arrow*` icon usages.** Lucide / Heroicons do **not** auto-flip. Each directional arrow needs `rtl:rotate-180` or an `<Arrow direction="end" />` wrapper.

## Why not blanket-codemod now

1. The risk-reward is poor: fixing 2,200 utilities to support a locale we haven't shipped invites regressions in production routes that are working fine today.
2. Phase 5 launch route (UAE Arabic) only touches 7 files. Codemod everything-everywhere first means weeks of regression testing on routes nobody is asking for.
3. Logical-property utilities (`ms-N`, `me-N`, `start-N`, `end-N`) work fine in LTR contexts too — the migration can be incremental, route-by-route, as RTL content lands.
